# SetUp IA + 1차 PRD (MVP)

## 1) 문서 목적
- `SetUp` 페이지를 운영 기준값 관리 허브로 정의한다.
- 현재 운영 중인 화면(`institutions`, `contacts`, `supports`, `people`)과 연결되는 관리 항목을 확정한다.
- 1차 구현 범위, 권한, 검증, 테스트 기준을 사전에 합의한다.

## 2) 현재 상태 요약
- 라우트 기준으로 운영 화면은 이미 존재한다: `people`, `institutions`, `contacts`, `supports`, `potential-institutions`.
- 사이드바에 `Setup` 메뉴 텍스트는 있으나, 전용 페이지/하위 정보 구조는 미정이다.
- `people` 기능에서 팀(부서) 생성/삭제와 자동 팀코드(`Axx`)가 이미 적용되어, SetUp의 조직 관리 영역 기반이 준비되어 있다.

---

## 3) IA 확정 (SetUp 하위 메뉴)
SetUp 1차 IA를 아래처럼 확정한다.

1. `조직/팀 관리`
2. `공통 코드 관리`
3. `권한/역할 관리`
4. `사용자 계정 관리`
5. `시스템 기본값`
6. `변경 이력`

> 1차 MVP는 `조직/팀 관리`, `공통 코드 관리`, `권한/역할 관리`, `변경 이력`을 우선 구현한다.

---

## 4) 화면 규칙 (MVP 공통 UX)
모든 SetUp 하위 화면은 아래 구조를 공통 사용한다.

- 상단: 화면 제목 + 설명 + 요약 지표(총 건수/활성 건수)
- 중단: 검색/필터/정렬 바
- 본문: 리스트 테이블
- 액션: `신규 생성`, `수정`, `삭제` (삭제는 확인 모달 필수)
- 모달: 생성/수정 모달을 동일 패턴으로 통일
- 상태: 로딩, 빈 데이터, 에러 메시지 표시 분리

### 4-1) 조직/팀 관리 화면 규칙
- 리스트 컬럼: 팀코드(DEPTNO), 팀명(DEPTNAME), 상위부서(ADMRDEPT), 위치(LOCATION), 소속 인원수
- 생성 규칙: 팀코드 자동 생성(`A01`, `A02`, ...)
- 삭제 규칙: 소속 인원이 있으면 삭제 차단
- 필수 검증: 팀명 길이, 팀코드 유니크(자동 생성 충돌 방지 포함)

### 4-2) 공통 코드 관리 화면 규칙
- 카테고리 탭: 직책, 고객유형, 상태값 등
- 각 카테고리 내 코드값 추가/수정/비활성 처리
- 사용 중인 값은 즉시 삭제 금지(비활성 우선)

### 4-3) 권한/역할 관리 화면 규칙
- 역할 목록: 관리자/일반/조회전용(기본)
- 메뉴 접근 매트릭스: `view/create/update/delete` 단위 체크
- 저장 시 역할별 권한 스냅샷 기록

### 4-4) 변경 이력 화면 규칙
- 필터: 기간, 사용자, 대상메뉴, 작업유형(CREATE/UPDATE/DELETE)
- 컬럼: 변경시각, 사용자, 대상, 변경 전/후 요약, IP(가능 시)

## 5) DB 매핑 (현재 테이블 기준)
SetUp에서 직접/간접으로 다루는 데이터 소스 매핑:

| SetUp 항목 | 1차 데이터 소스 | 연결 화면/기능 |
|---|---|---|
| 조직/팀 관리 | `department` (`DEPTNO`, `DEPTNAME`, `ADMRDEPT`, `LOCATION`) | `people` 팀 필터/사이드바 |
| 사용자 계정/직원 | `employee` (`EMPNO`, `WORKDEPT`, `JOB`, `STATUS`, `EMAIL`) | `people` 직원 목록/수정 |
| 기관 담당자/고객유형 | `S_Account_Information` (`TR`, `CS`, `CO`, `Customer_Type`) | `institutions`, `contacts` |
| 기관 기본정보 | `S_AccountName` (`Gubun`, `GSno` 등) | `institutions` |
| 지원 이력 운영 상태 | `S_SupportInfo_Account` (지원 내역 필드) | `supports`, 상세 모달 |

> 참고: 공통 코드(직책/상태/고객유형)는 1차에 기존 테이블의 distinct 값 기반으로 운영하고, 2차에서 전용 코드 테이블 분리를 고려한다.

## 6) RBAC 정책 초안

### 역할 정의
- `admin`: SetUp 전체 접근 + 생성/수정/삭제 가능
- `manager`: 조회 + 일부 수정 가능(삭제 제한)
- `viewer`: 조회만 가능

### 정책 원칙
- 메뉴 노출 제어 + 서버 액션 제어를 동시에 적용한다.
- UI 버튼 숨김만으로 끝내지 않고, 저장/삭제 메서드에서도 권한 체크를 강제한다.
- 위험 액션(삭제/권한 변경)은 확인 모달 + 이력 기록을 필수로 한다.

### 1차 권한 매트릭스
| 메뉴 | admin | manager | viewer |
|---|---|---|---|
| 조직/팀 관리 | CRUD | R,U | R |
| 공통 코드 관리 | CRUD | R,U | R |
| 권한/역할 관리 | CRUD | R | - |
| 변경 이력 | R | R | R |

## 7) 구현 우선순위 + 테스트 체크리스트

## 우선순위
1. 조직/팀 관리
2. 공통 코드 관리
3. 권한/역할 관리
4. 변경 이력

## 테스트 체크리스트
- [ ] 팀 생성 시 코드가 자동 증가(`Axx`)한다.
- [ ] 팀 삭제 시 소속 직원이 있으면 차단된다.
- [ ] SetUp 변경이 `people`/`institutions` 화면에 즉시 반영된다.
- [ ] 관리자/매니저/조회전용 권한 차이가 실제 액션에서 동작한다.
- [ ] 삭제/권한변경이 변경 이력에 남는다.
- [ ] 빈 데이터/오류 상태에서 사용자 안내가 명확하다.

---

## 8) 범위 구분

### In scope (1차)
- 조직/팀 관리
- 공통 코드 관리(핵심 카테고리)
- 역할/권한 기본 정책
- 변경 이력 조회

### Out of scope (2차)
- 외부 API 키/연동 설정
- 백업/복구 자동화
- 고급 알림 정책

---

## 9) 완료 기준
- SetUp IA가 화면/라우트/권한 정책과 일치한다.
- MVP 4개 영역에 대한 데이터 소스와 검증 규칙이 문서화되어 있다.
- 개발 시작 시 추가 해석 없이 구현 가능한 수준의 체크리스트를 제공한다.
